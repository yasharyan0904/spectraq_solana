‚ĚĮ What is the use of WSOL? And what is                                          
  WSOL_MINT=So11111111111111111111111111111111111111112 ?                       
  ‚éŅ ¬†‚ßČ Selected 1 lines from .env in Visual Studio Code

‚óŹ What wSOL is

  SOL ‚Č† an SPL token. Native SOL lives as raw lamports in any Solana account's
  header ‚ÄĒ it's the gas token, owned by the System Program. SPL infrastructure
  (token transfers, ATAs, swap programs, AMMs, vaults like ours) only knows how
  to move SPL tokens. So if you want to use SOL inside an SPL context, you have
  to wrap it.

  wSOL = an SPL token mint that 1:1 represents native SOL. You "wrap" by
  transferring native SOL into a wSOL token account, then calling
  spl_token::sync_native which credits the SPL token amount based on the
  lamports balance. To "unwrap," you close the wSOL account and the lamports
  flow back to the owner.

  What So11111111111111111111111111111111111111112 is

  That's the canonical wSOL mint address ‚ÄĒ hardcoded across the entire Solana
  ecosystem (SPL Token program, Jupiter, Raydium, Phantom, Solana docs, etc).
  It's a vanity address: encodes the string "So1‚Ä¶2", base58. Nobody holds the
  private key; you can't change its mint authority. Total supply = total wrapped
   SOL across all wSOL ATAs at any moment. Always 9 decimals (matches lamports).

  Why SpectraQ uses it

  In our vault:
  - sol_vault is an SPL token account whose mint is wSOL. That lets the program
  use the same token::transfer CPI for both USDC and SOL legs.
  - When Jupiter routes a swap that ends in "SOL," it actually delivers wSOL to
  your destination ATA (it's an SPL operation, the program can't deliver native
  lamports). Same on the input side: a SOL‚ÜíUSDC swap consumes wSOL.
  - Pyth's SOL/USD feed prices native SOL ‚ÄĒ wSOL trades 1:1 with it, so the same
   feed works.

  Caveat for prompt 1 vs later prompts

  The lifecycle test you just saw uses a fake locally-created mint as a stand-in
   for wSOL ‚ÄĒ the program doesn't require it to be the real So11‚Ä¶112, it just
  stores whatever sol_mint you pass at initialize_vault. The real wrap flow
  (SystemProgram.transfer ‚Üí sync_native) lands when we initialize the production
   vault in a later prompt. At that point WSOL_MINT in .env becomes load-bearing
   ‚ÄĒ initialize_vault will be called with that mint, and Jupiter/Pyth
  integration assumes it.
